Solid support is additive. If an API is not exported by `@better-auth-ui/solid`, do not document it as package support.

## Package-level non-goals

| Area | Current boundary |
| --- | --- |
| React email templates | `@better-auth-ui/solid` does not export email templates and does not depend on `@react-email/components`. Render Solid-compatible email HTML from app/server code if you need it. |
| Toast UI | The package exposes query/mutation factories only. Toast rendering belongs to the app or copied UI files. |
| HeroUI React components | HeroUI is React-only. Solid users should stay with Solid package APIs or install Solid-owned copied components through Zaidan. |
| React render-hook internals | Solid docs cover Solid Query options and Solid components, not React hook mechanics. |
| Captcha UI | There is no Solid package widget/token lifecycle yet. Treat it as a future design task. |
| Docs runtime execution | The docs app hosts content and static assets. Solid source executes in the Solid Start example, not inside the React/Fumadocs runtime. |

## Email rendering in Solid

For a small Solid-owned email template, render the component to HTML on the server with `solid-js/web`:

```tsx
import { renderToString } from "solid-js/web"

function WelcomeEmail(props: { name: string }) {
  return (
    <html>
      <body>
        <h1>Welcome, {props.name}</h1>
        <p>Thanks for joining.</p>
      </body>
    </html>
  )
}

const html = renderToString(() => <WelcomeEmail name="Andrés" />)
```

Pass the resulting `html` string to your mail provider from app/server code. Keep that rendering choice app-owned unless a future design explicitly adds a Solid email package surface.

## Before adding a shim

Open a design task first. Decide the owning package, runtime dependency, API shape, and test strategy before adding compatibility wrappers.
