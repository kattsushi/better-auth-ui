The Solid runtime integration is simple: create one Better Auth client, provide it through `AuthProvider`, and share the same Solid Query client used by your TanStack Start app.

## Quick path

1. Create the auth client in app code.
2. Create or reuse your `QueryClient`.
3. Put `AuthProvider` inside your route shell so all auth UI reads the same context.

```tsx
import { AuthProvider, createAuthClient } from "@better-auth-ui/solid"
import { QueryClient } from "@tanstack/solid-query"
import type { JSX } from "solid-js"

const authClient = createAuthClient({ baseURL: "/api/auth" })
const queryClient = new QueryClient()

export function Providers(props: { children: JSX.Element }) {
  return (
    <AuthProvider authClient={authClient} queryClient={queryClient}>
      {props.children}
    </AuthProvider>
  )
}
```

## Provider primitive

`AuthProvider` accepts the same Solid auth config used by the copied UI components and runtime helpers. Its important runtime behavior is:

| Concern | Behavior |
| --- | --- |
| Auth context | `useAuth()` reads the resolved config from Solid context. |
| Query client | `queryClient` is optional; the package has a fallback client, but app-owned clients are preferred. |
| Children | `children` may be a Solid element or function; function children are evaluated while the auth config is available. |
| Errors | Package query/mutation helpers preserve Better Auth errors by passing `throw: true`. |

## TanStack Start boundary

Keep the server route (`/api/auth/*`) in your app's TanStack Start server code. The Solid package does not own route generation; it provides helpers used by the route shell and auth components.
